

## 1. 项目背景

本项目目标是实现一个基于 ROS2 的智能跟拍机器人。系统由移动底盘、云台相机、深度相机、边缘计算平台和语音/遥控交互模块组成，能够完成目标识别、自主跟随、画面稳定跟拍、遥控/自主模式切换等功能。

项目中的核心控制问题是：

> 当目标在画面中发生偏移时，云台和底盘都可以参与修正，那么应该如何分配二者的运动？

例如，目标出现在画面左侧时：

- 云台 yaw 可以向左转，使目标回到画面中心；
    
- 底盘也可以向左转，使机器人整体朝向目标；
    
- 如果二者没有协调，可能出现过冲、振荡、画面抖动或云台打到机械极限等问题。
    

因此，本项目需要设计一套面向智能跟拍机器人的云台-底盘协同控制方案。

---

## 2. 总体技术路线

本项目不直接采用复杂的 MPC 或完整 PBVS 控制，而应采用渐进式路线：

```text
V1：基础 MVP 跟随版
    云台追图像误差
    底盘追距离和云台偏角

V2：稳定视觉伺服版
    加入滤波、死区、限幅、目标锁定、丢失处理

V3：混合 IBVS-PBVS 协同版
    云台采用 IBVS
    底盘采用 PBVS
    协同层负责控制分配

V4：MPC / 优化协同控制版
    将云台和底盘作为整体系统统一优化
```

总体原则是：

> 云台负责快，底盘负责稳；  
> 云台负责画面，底盘负责位置；  
> 云台短期跟踪，底盘长期对齐。

## 3. 阶段 V1：基础 MVP 跟随版

### 3.1 阶段目标

V1 阶段的目标是先让系统跑起来，实现最小可行闭环：

```text
人出现在画面中
    ↓
YOLO 检测目标
    ↓
获取目标中心点和深度
    ↓
云台让目标居中
    ↓
底盘保持跟随距离
    ↓
底盘根据云台偏角慢速转向
```

该阶段不追求理论最优，而是追求系统可运行、可调试、可展示。

---

### 3.2 控制算法选择

V1 阶段推荐采用：

```text
云台：基于图像误差的简化 IBVS / P 控制
底盘：基于目标深度的距离控制
底盘转向：基于云台 yaw 偏角的慢速补偿控制
```

不要在第一版中直接写成：

```text
云台用 IBVS
底盘用 PBVS
```

更合理的说法是：

```text
云台使用图像误差控制实现目标居中；
底盘使用距离误差和云台偏角实现低频跟随；
allocation_node 统一完成云台和底盘的协同分配。
```

---

### 3.3 控制输入

V1 阶段需要以下输入：

```text
cx：目标 bbox 中心点 x 坐标
cy：目标 bbox 中心点 y 坐标
W：图像宽度
H：图像高度
Z：目标深度
Z_desired：期望跟随距离
q_yaw：云台当前 yaw 角
q_pitch：云台当前 pitch 角
```

归一化图像误差：

```text
e_x = (cx - W / 2) / (W / 2)

e_y = (cy - H / 2) / (H / 2)
```

距离误差：

```text
e_z = Z - Z_desired
```

---

### 3.4 推荐控制律

云台 yaw 控制：

```text
gimbal_yaw_vel = -Kx * e_x
```

云台 pitch 控制：

```text
gimbal_pitch_vel = -Ky * e_y
```

底盘前后控制：

```text
base_vx = Kz * e_z
```

底盘转向控制：

```text
base_wz = Kb * q_yaw
```

其中最关键的是：

> 底盘不要直接追图像水平误差 `e_x`，而应优先追云台 yaw 偏角 `q_yaw`。

这样可以形成自然的快慢分工：

```text
目标短时偏移：
    云台快速修正，保证画面稳定

目标长期在一侧：
    云台 yaw 偏角逐渐变大
    底盘开始慢速转向
    云台逐渐回到中位
```

---

### 3.5 V1 阶段意义

V1 阶段的意义是：

1. 快速完成项目闭环，验证硬件链路是否可用；
    
2. 验证相机、YOLO、深度、云台和底盘之间的数据流；
    
3. 建立后续升级所需的 ROS2 节点框架；
    
4. 初步实现云台-底盘协同，而不是两个执行器各自独立控制；
    
5. 为比赛或项目验收提供最低可展示版本。

## 4. 阶段 V2：稳定视觉伺服版

### 4.1 阶段目标

V1 跑通后，系统通常会出现以下问题：

```text
目标框抖动
深度值跳变
多人场景目标切换
目标短时丢失
底盘小幅抖动
云台频繁微调
速度命令突变
```

V2 阶段的目标不是更换高级算法，而是提高系统稳定性和展示效果。

---

### 4.2 增加目标管理模块

需要加入 `target_manager_node`，负责目标选择、锁定和丢失处理。

目标管理逻辑：

```text
如果当前 target_id 仍然存在：
    继续跟随当前目标

如果当前 target_id 短时丢失：
    保持上一帧状态或减速等待

如果目标丢失超过阈值：
    进入搜索模式

如果用户重新指定目标：
    切换跟随目标
```

该模块可以避免多人场景中目标频繁切换。

---

### 4.3 增加滤波

建议对以下变量进行滤波：

```text
e_x：水平图像误差
e_y：垂直图像误差
Z：目标深度
q_yaw：云台 yaw 角
base_vx：底盘线速度命令
base_wz：底盘角速度命令
```

可先采用一阶低通滤波：

```text
x_filtered = α * x_new + (1 - α) * x_old
```

推荐参数范围：

```text
图像误差 e_x, e_y：
    α = 0.4 ~ 0.7

深度 Z：
    α = 0.2 ~ 0.5

底盘速度命令：
    α = 0.2 ~ 0.4
```

---

### 4.4 增加死区

为了避免系统在目标接近中心时仍然频繁微调，需要加入死区：

```text
|e_x| < 0.05：
    云台 yaw 不动

|e_y| < 0.05：
    云台 pitch 不动

|Z - Z_desired| < 0.2 m：
    底盘前后不动

|q_yaw| < 5°：
    底盘不转
```

死区可以显著减少画面抖动和底盘小幅振荡。

---

### 4.5 增加速度限幅和加速度限幅

除了限制最大速度，还应限制速度变化率。

例如：

```text
base_vx 不应从 0 突然跳到 0.8 m/s
base_wz 不应瞬间产生大角速度
gimbal_yaw_vel 不应快速反复改变方向
```

可以增加：

```text
max_base_vx
max_base_wz
max_gimbal_yaw_vel
max_gimbal_pitch_vel

max_delta_base_vx
max_delta_base_wz
max_delta_gimbal_yaw_vel
max_delta_gimbal_pitch_vel
```

速度限幅保证安全，加速度限幅保证画面平滑。

---

### 4.6 增加云台回中策略

V2 阶段可以在云台控制中加入较小的回中项：

```text
gimbal_yaw_vel = -Kx * e_x - Kq * q_yaw
```

其中：

```text
-Kx * e_x：
    负责让目标居中

-Kq * q_yaw：
    负责让云台不要长期偏离中位
```

需要注意：

```text
Kq 不宜过大
否则云台会急于回中，影响目标居中效果
```

---

### 4.7 V2 阶段意义

V2 阶段的意义是：

1. 让系统从“能跑”变成“稳定可展示”；
    
2. 解决目标框抖动、深度跳变和底盘抖动问题；
    
3. 解决多人场景下目标频繁切换问题；
    
4. 提升跟拍画面的平滑度；
    
5. 为后续 PBVS 和 MPC 提供稳定输入。

## 5. 阶段 V3：混合 IBVS-PBVS 协同版

### 5.1 阶段目标

V3 阶段开始引入更正式的视觉伺服结构：

```text
云台：IBVS
底盘：PBVS
协同层：控制分配
```

该阶段的核心目标是：

> 云台负责图像平面内的目标居中，底盘负责机器人与目标之间的空间相对位置控制。

---

### 5.2 云台采用 IBVS

云台的任务是让目标保持在画面中心，因此非常适合使用 IBVS。

云台控制输入：

```text
e_x：目标水平图像误差
e_y：目标垂直图像误差
```

云台控制输出：

```text
gimbal_yaw_vel
gimbal_pitch_vel
```

控制目标：

```text
e_x → 0
e_y → 0
```

也就是：

```text
目标偏左/偏右：
    云台 yaw 修正

目标偏上/偏下：
    云台 pitch 修正
```

---

### 5.3 底盘采用 PBVS

底盘的任务是控制机器人与目标之间的空间相对位置，因此更适合 PBVS。

需要先计算目标在相机坐标系下的 3D 点：

```text
P_camera = [X_c, Y_c, Z_c]
```

然后通过 TF 转换到底盘坐标系：

```text
P_base = [X_b, Y_b, Z_b]
```

对移动底盘来说，主要使用：

```text
X_b：目标在机器人前方的距离
Y_b：目标相对机器人的横向偏移
```

底盘线速度：

```text
base_vx = Kx * (X_b - X_desired)
```

底盘角速度：

```text
base_wz = Kyaw * atan2(Y_b, X_b)
```

相比 V1 阶段，V3 的底盘控制更加明确：

```text
V1：
    底盘基于目标深度和云台偏角控制

V3：
    底盘基于目标在 base_link 下的空间位置控制
```

---

### 5.4 协同层控制分配

即使云台采用 IBVS、底盘采用 PBVS，也不能让二者完全独立。

原因是：

```text
目标水平偏移既可以由云台 yaw 修正
也可以由底盘 yaw 修正
```

因此仍然需要 `allocation_node` 或 `coordination_node` 负责：

```text
云台优先快速居中
底盘慢速调整机身朝向
避免云台接近机械极限
避免底盘过度转向
控制速度和加速度
处理目标丢失与安全停止
```

---

### 5.5 V3 阶段意义

V3 阶段的意义是：

1. 从经验规则控制升级为更正式的视觉伺服框架；
    
2. 云台和底盘分工更加清晰；
    
3. 云台使用图像误差实现画面稳定；
    
4. 底盘使用空间位姿实现更合理的跟随路径；
    
5. 为后续优化控制和 MPC 建立数学模型基础。

## 6. 阶段 V4：MPC / 优化协同控制版

### 6.1 阶段目标

V4 是高级升级阶段，目标是将云台和底盘视为一个整体冗余执行器系统，统一进行控制分配。

该阶段需要解决的问题是：

```text
目标图像误差要小
跟随距离要合适
云台不要接近机械极限
底盘速度不要太大
云台速度不要太大
画面运动要平滑
控制量变化不能突变
```

---

### 6.2 优化型控制分配

在进入完整 MPC 之前，可以先做当前时刻的优化型控制分配。

控制量：

```text
u = [
    base_vx,
    base_wz,
    gimbal_yaw_vel,
    gimbal_pitch_vel
]
```

优化目标可以设计为：

```text
J =
    w1 * e_x²
  + w2 * e_y²
  + w3 * e_z²
  + w4 * q_yaw²
  + w5 * base_wz²
  + w6 * gimbal_yaw_vel²
```

含义：

```text
e_x², e_y²：
    目标尽量居中

e_z²：
    跟随距离尽量合适

q_yaw²：
    云台尽量接近中位

base_wz²：
    底盘不要过度转向

gimbal_yaw_vel²：
    云台运动不要过猛
```

约束条件：

```text
base_vx_min ≤ base_vx ≤ base_vx_max
base_wz_min ≤ base_wz ≤ base_wz_max

gimbal_yaw_vel_min ≤ gimbal_yaw_vel ≤ gimbal_yaw_vel_max
gimbal_pitch_vel_min ≤ gimbal_pitch_vel ≤ gimbal_pitch_vel_max

q_yaw_min ≤ q_yaw ≤ q_yaw_max
q_pitch_min ≤ q_pitch ≤ q_pitch_max
```

---

### 6.3 MPC 协同控制

MPC 相比普通优化分配，多了“预测未来”的能力。

MPC 状态可以设计为：

```text
x = [
    e_x,
    e_y,
    e_z,
    q_yaw,
    q_pitch,
    base_vx,
    base_wz
]
```

控制量：

```text
u = [
    gimbal_yaw_vel,
    gimbal_pitch_vel,
    base_vx_cmd,
    base_wz_cmd
]
```

MPC 目标函数：

```text
minimize:
    图像水平误差
  + 图像垂直误差
  + 距离误差
  + 云台偏离中位惩罚
  + 底盘速度惩罚
  + 云台速度惩罚
  + 控制量变化率惩罚
```

约束：

```text
云台 yaw 角限制
云台 pitch 角限制
云台角速度限制
底盘线速度限制
底盘角速度限制
安全距离限制
控制量变化率限制
```

---

### 6.4 V4 阶段意义

V4 阶段的意义是：

1. 将云台和底盘作为整体系统统一控制；
    
2. 显式处理速度、角度、安全距离等约束；
    
3. 在目标居中、跟随距离、云台回中和底盘平滑之间做权衡；
    
4. 可以提前预测云台是否接近极限，让底盘提前参与；
    
5. 适合作为项目的高级创新点、论文亮点或比赛答辩重点。

## 10. 最终推荐实施顺序

推荐按照以下顺序实施：

```text
第一步：
    完成 V1 基础 MVP
    重点是让系统闭环跑起来

第二步：
    完成 V2 稳定性增强
    重点是滤波、死区、限幅、目标锁定

第三步：
    完成 V3 混合 IBVS-PBVS

    重点是控制分配和约束优化
```

不要跳过 V1 和 V2 直接做 V4。对于智能跟拍机器人来说，感知稳定性、执行器延迟、深度噪声和目标管理问题往往比控制算法本身更容易影响最终效果。

---

## 11. 项目书中可使用的技术路线表述

本项目拟采用分阶段递进式技术路线。首先构建基于图像误差的云台视觉伺服与基于深度信息的底盘跟随控制，实现智能跟拍机器人的最小可行系统。在此基础上，引入目标锁定、误差滤波、死区控制、速度限幅和目标丢失处理机制，提高系统稳定性和实际展示效果。随后，利用深度相机和 TF 坐标变换估计目标在机器人底盘坐标系下的三维位置，将底盘控制升级为基于位置的视觉伺服方法，实现更准确的空间跟随。最终，将云台和底盘视为冗余多执行器系统，进一步引入基于约束优化或 MPC 的控制分配方法，在目标居中、跟随距离、云台回中、底盘平滑性和执行器约束之间实现统一优化。

---

## 12. 一句话总结

本项目的控制系统升级路线可以概括为：

```text
先用“云台追图像误差，底盘追距离和云台偏角”跑通；
再用滤波、死区、限幅和目标锁定把系统做稳；
然后把底盘升级为基于目标 3D 位置的 PBVS；
最后用 QP/MPC 实现云台和底盘的统一协同优化。
```
